# Сценарии работы

## 1. Гибридный (рекомендуемый для сложных архитектурных задач)
- **Архитектор** – браузерный DeepSeek-R1 (или другой LLM вне Roo Code). Генерирует план, структуру, сохраняет в `memory-bank/` или в Markdown.
- **Исполнитель** – Roo Code (режим Code). Читает план из `memory-bank/` (через MCP или вручную), использует нужные MCP, пишет код.
- **Плюсы**: нет перегрузки контекста, модель не вязнет в MCP-инструментах, можно использовать любые MCP по требованию.
- **Минусы**: ручное копирование плана, нет единого окна.

## 2. Автономный (всё в Roo Code)
- **Архитектор** – режим Architect в Roo Code/Kilo code/терминальный агент CLI с **минимальным набором MCP** . Использовать локальную модель (можно локальные модели).
- **Код** – режим Code с полным набором MCP (postgres, context7).
- **Плюсы**: всё в одном интерфейсе, автоматическая передача контекста.
- **Минусы**: нужно вручную отключать лишние MCP перед переключением режимов, чтобы не забивать контекст.

## Рекомендации
- Для сложных задач с большим количеством инструментов – гибридный.
- Для простых итераций – автономный.
- Всегда фиксируй принятые решения в `decisionLog.md` (в memory-bank).

## Бэкапы изменений

Перед изменением файла агент:
1. Создаёт `.backup/` в корне проекта (если нет).
2. Копирует исходный файл в `.backup/` с именем `<оригинал>.YYYYMMDD_HHMMSS.bak`.
3. После успешного выполнения задачи (код проверен в песочнице) бэкап **удаляется**.
4. Если задача не выполнена — бэкап остаётся.

## Логирование

Агент ведёт лог в `logs/` проекта.
- Формат: `logs/agent_YYYYMMDD_HHMMSS.log`
- Записывать: команды, ошибки, результаты выполнения (особенно если вывод не копируется).

## Управление контекстом и экономия токенов

- **Roo Code / Kilo Code** загружают все глобальные и локальные правила автоматически. Для экономии отключай неиспользуемые MCP (см. `02-mcp-usage.md`).
- **Терминальный агент**:
  - Перед началом читает только `memory-bank/activeContext.md`.
  - При необходимости подгружает конкретные файлы правил из `~/.roo/rules/` или `.roo/rules/`.
  - Использует команды очистки контекста (например, `/clear`), если сессия засорилась.

## Обновление глобальных правил

Если в проекте меняется системная архитектура (новые контейнеры, пути, скрипты):
- **Терминальный агент** самостоятельно вносит изменения в `~/.roo/rules/`.
- **Roo Code / Kilo Code** запрашивают у пользователя включение MCP `filesystem` и предлагают обновить файлы. Если пользователь отказывается, задача передаётся терминальному агенту.

## Терминальный агент (CLI)

Для единой работы с глобальными правилами терминальный агент должен быть настроен на чтение файлов из `~/.roo/rules/`.

### Настройка opencode
- Создать `~/.config/opencode/opencode.json` с секцией `instructions`, перечисляющей все файлы правил (полные пути).
- Пример конфигурации см. в `99-reminders.md`.

### Проверка
Запустить с тестовой инструкцией:
```bash
echo "Ты должен ответить только словом КУКУ" > /tmp/test.md
opencode --instructions /tmp/test.md -p "Привет" --model siliconflow/zai-org/GLM-5

 ### Обновление memory‑bank (шаги, ретроспектива)

После каждого значимого этапа (завершённая задача, изменение архитектуры, обновление правил) агент **обязан** обновить:
- `memory-bank/activeContext.md` (что сейчас делается)
- `memory-bank/progress.md` (что сделано)
- при необходимости `decisionLog.md` (принятые решения)

Обновление происходит **без напоминания** — это обязательное действие.
